/goal G-PORT — Port slack-flue to Flue behind a behavioral parity harness, then delete the custom harness.

Context: slack-flue has two stacks: a working hand-rolled harness (src/server.ts → src/slack/events-app.ts → src/runtime/slack-thread-runner.ts, plus its own provider registry, session store, dedupe) and a vestigial Flue lane (src/agents/slack-thread.ts) that throws at init from a bad model specifier. Decision: the product runs on Flue. The 46 tests drive signed Slack fixtures through the app with an injected recording fetch — they encode the behavior to preserve. Flue provides `flue add channel slack`, registerProvider() (incl. openai-compatible baseUrl registrations usable as a local offline stub), durable agents (db.ts adapter), and `flue dev --target node`. Flue's Slack docs assign admission filtering and duplicate-claiming to the app, so turn-normalization.ts survives verbatim. Time-box ~2–3 weeks; old lane frozen to bug-fixes-only.

Stage 0 — Extract the contract. Build a lane-agnostic scenario suite: signed fixture(s) in → observables out (reply or nothing on the wire; count/target of finals recorded by a fake-Slack backend; status set then cleared; reply-content properties; conversations.replies/history params; N duplicate events → 1 reply). Sources: the 46 tests + docs/play-slack.md §5. Assertions behavioral, never byte-exact call sequences. Lane A adapter = createSlackEventsApp + injected fetch. VERIFY: suite green on main; mutation check — break one behavior (e.g. skip the dedupe claim) and exactly the right scenario goes red; npm test stays green.

Stage 1 — Boot the Flue lane. Fix src/agents/slack-thread.ts:17 to 'cloudflare-workers-ai/@cf/zai-org/glm-5.2' (re-confirm vs lockfile runtime); add engines >=22.18 + .nvmrc. VERIFY: a resolver-driving test is red on main, green after; `npm run flue:build` green in CI on pinned Node.

Stage 2 — Stand up the Flue Slack surface. `flue add channel slack`; port app policy into the channel handler: normalizeSlackTurn admission unchanged; duplicate claim keyed (channel, message ts) + event_id in an app-owned claim store; assignment/config; bounded hydration via @slack/web-api WebClient; Assistant status + chatStream. Slack API base URL injectable (slackApiUrl env; fallback: global-fetch preload). registerProvider: anthropic, cloudflare-workers-ai, local deterministic stub. VERIFY: `flue dev --target node` serves /channels/slack/events (signed → 200, tampered → 401, url_verification echoed); one full offline turn vs the stub with network recorded/blocked — zero external traffic.

Stage 3 — Parity. Lane B adapter: spawn the Flue app, WebClient → local fake Slack, quiescence-poll the wire log (dispatch is async). VERIFY: the entire suite passes on Flue unmodified; intentional differences go in a reviewed per-scenario exceptions file, never silent edits; CI runs both lanes during coexistence.

Stage 4 — Flue-only gains. VERIFY with committed evidence: (a) kill mid-conversation with file-backed db.ts → follow-up retains context (transcript attached); (b) same fixture answered via anthropic/… and cloudflare-workers-ai/… (both replies attached); (c) lookup_channel_brief as one defineTool, invoked by the model, deny enforced with honest message; (d) suite still green.

Stage 5 — Delete the harness. Remove slack-thread-runner, events-app, server.ts, web-api-replies, providers/, session-store, old dedupe, safe-tools + internals-only tests; keep turn-normalization, config/, fixtures, suite. VERIFY: suite green AFTER deletion; npm test + flue:build green; grep shows no imports of deleted modules; README/play-slack.md lead with flue dev; LOC delta reported.

Done: CI runs the suite offline (zero credentials) on every PR; dated port record in docs/decisions/ with parity evidence, exceptions file, Stage-4 artifacts. Every proof produced by running commands, not code reading.
