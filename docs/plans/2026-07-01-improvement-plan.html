slack-flue · state of the prototype & improvement plan · July 1, 2026

Two lanes.
One decision.

slack-flue answers Slack mentions with an AI agent. Today it does this through a hand-rolled harness that works and is well-tested — while the framework it was built to evaluate sits in a second, broken lane that has never actually been tried. Everything in this plan flows from closing that gap deliberately instead of by accident.

Produced by a four-phase multi-agent review: 8 subsystem readers → 10 hands-on scenario testers (real servers, signed fixtures, recorded wire traffic) → 21 adversarial verifiers (independent re-reproduction; 31/31 findings confirmed, 1 sub-claim refuted, 7 severities corrected) → 3 plan critics (a Flue skeptic, an OSS maintainer, a prototype-purpose auditor) whose objections reshaped the priorities below.

Updated 2026-07-01: the owner made the call — the product runs on Flue. Tier 2 is no longer a decision spike; it is a gated behavioral-parity port (/goal G-PORT). The old decision scorecard survives as port acceptance criteria (§07).

Lane A · hand-rolledRUNS TODAY

src/server.ts → events-app → slack-thread-runner. 2.9k lines, 46 green tests, fully offline-testable. But: no real tool loop (a keyword substring fires one fixed tool), no conversation memory, a closed 2-provider union, and all state in per-process Maps.

Lane B · Flue / CloudflareBROKEN ON ARRIVAL

defineAgent + flue build. Ships an invalid model specifier (throws at init) and a built Worker with zero working endpoints. Yet Flue itself offers exactly what Lane A lacks: durable conversations, an open provider registry, a real tool loop, and an official Slack channel.

Decision made · gated port, time-boxed ≈2–3 weeks

The product runs on Flue. A lane-agnostic scenario suite extracted from the existing tests becomes the port's contract: Lane A passes it today, the Flue lane must pass it unmodified, and only then is the custom harness deleted — with the suite still green after deletion.

01Executive summary

What we found, what we recommend, and what to do first.

The system works. The primary loop — signed Slack event in, status shown, context hydrated, model reply streamed back into the thread — passed hands-on testing end to end, offline, across mentions, DMs, App Home, thread continuations, duplicates, and malformed input. The test suite is fast (2.2 s) and the core has genuinely good seams (injectable fetch, sink/context/provider interfaces).

But the project is quietly failing its own charter. docs/START_HERE.md defines this repo as a prototype for evaluating Flue, with explicit stop/pivot gates. Verified reality: the tested stack contains no Flue, the Flue stack contains no Slack and has never executed successfully (invalid model specifier; a deployable artifact serving only 404s), and the "same task on Claude and a non-Claude model" gate is unproven because the Claude lane is a hardcoded stub. Hardening the hand-rolled harness further is not a neutral default — it is the charter's stop condition, enacted by accident.

Meanwhile two high-severity correctness bugs are live in the lane that runs, and both are independent of the framework decision: any other bot that mentions this bot triggers full agent turns (a bot-to-bot loop risk with no rate limiter anywhere), and one human mention inside an active thread arrives as two Slack events and produces two full replies.

Recommendation, in order:

  1. Tier 0 (same day): five small, lane-invariant unblocks — fix the Flue specifier, pin Node, validate env at boot, add one structured log line per turn, stand up CI, add a README. Hours each.
  2. Tier 1 (1–2 days): kill both high-severity admission bugs with regression tests converted from this review's repro drivers.
  3. Tier 2 (≈2–3 weeks, time-boxed): execute /goal G-PORT — the behavioral-parity port to Flue. Extract the existing tests' expectations into a lane-agnostic scenario suite, make the Flue lane pass it unmodified, prove the Flue-only gains (restart durability, provider swap, model-invoked tools), then delete the custom harness with the suite still green.
  4. Tiers 3–4 (post-port): the power pack (memory, tool loop, durability, resilience, output safety) and the contributor platform (offline demo loop, four-seam CONTRIBUTING, good-first-issue pipeline) — built once, on the Flue lane only.

Five of these items carry full /goal statements written to be handed off and verified without further planning.

One operational note: the CLOUDFLARE_API_TOKEN in .env.slack.local is invalid (verified 401 against Cloudflare's token-verify endpoint on 2026-07-01). Live Workers AI runs are impossible until it is re-minted.

02How this was verified

Every claim below survived independent, adversarial reproduction. Nothing here rests on code reading alone.

  • Phase 1 — Understand. Eight parallel readers mapped the subsystems (ingress, admission, context, runtime, presentation, providers/config/tools, Flue usage, test/doc coverage), each returning flows, coupling points, extension-point pain, and suspected risks with file:line evidence.
  • Phase 2 — Exercise. Ten testers drove the real system: the actual src/server.ts entrypoint plus a fake-Slack HTTP rig built on the app factory's injectable fetch. 87 checks across happy paths, admission negatives, fan-out/dedupe, restarts and dual instances, signature hardening, context-window matrices, formatting and streaming, provider failure matrices against a local stub, the Flue build lane, and a fresh-contributor dry run. All with fake Slack credentials; zero traffic reached the real workspace.
  • Phase 3 — Refute. Twenty-one verifiers, each instructed that the finding was wrong until their own fresh reproduction proved otherwise, re-reproduced every finding from scratch and hunted for rig artifacts. Result: 31/31 confirmed; the DM half of the double-post claim was refuted (Slack never sends app_mention in DMs); seven severities corrected, both up and down.
  • Phase 4 — Critique. Three critics attacked the draft plan itself. Their material changes: the two high bugs moved ahead of the framework bet (they are lane-invariant), the open-ended "convergence slice" became a time-boxed spike with a pass/fail scorecard, "durable state for free" was re-costed as real work on the Node lane, and the contributor story gained a license/governance floor and a fourth seam (context-sources/domain packs).

03Two lanes, one decision

The strategic question, both sides steelmanned, and how the plan resolves it.

The case for converging on Flue

  • The charter demands it. START_HERE goal 5: "Flue owns the agent harness/session loop." Its stop gate: "Slack/channel policy must be custom-built outside Flue with little remaining harness benefit." Building a tool loop, durable sessions, and an open provider registry by hand is that stop condition.
  • The gaps map one-to-one. Lane A's four structural ceilings — no model-driven tools, no conversation memory, closed provider union, process-local state — are precisely Flue's shipped competencies: defineTool, durable Durable-Object-backed conversation streams, registerProvider() (Anthropic, Workers AI bindings, Ollama, gateways in one call), and flue dev for both Node and Cloudflare targets.
  • The damning evidence is self-inflicted. The broken specifier is a one-line fix (the verifier proved cloudflare-workers-ai/@cf/zai-org/glm-5.2 resolves against the installed runtime), and the empty Worker exists because flue add channel slack — Flue's official Slack ingress blueprint — was never run. The lane failed because it was never seriously attempted, not because it was attempted and failed.
  • Seam economics decide the contributor goal. Measured in the dry run: adding one tool today touches 6 files across both lanes including a hardcoded keyword gate; adding a provider touches 8 files across a closed union. On Flue: one defineTool, one registerProvider call — and 16 more channel blueprints (Discord, Teams, GitHub, Telegram…) as the integration surface this project would otherwise have to invent.

The case for keeping the hand-rolled lane (the skeptic's brief)

  • Convergence fixes neither high-severity bug. Flue's own Slack docs assign bot-filtering, admission, and duplicate-claiming to the application. The rewrite spends effort on the part that already works while the broken part stays app-owned in both worlds.
  • The team could not see Flue failing. The lane shipped 100%-broken twice over and nobody noticed: flue build succeeded while producing a dead artifact, and flue run --target cloudflare swallowed the real error behind a generic 500. A framework you cannot observe failing is a framework you cannot yet operate.
  • Beta churn, stacked three deep. @flue 1.0.0-beta.8 (pinned exact), a Slack Web API "v8 release candidate," a Node ≥22.18 floor the default toolchain fails — and a runtime flag literally named temporaryLocalExposure on the exact surface that broke.
  • "Durable for free" is oversold. Per Flue's own docs, the Node target without a db.ts adapter loses all conversations on restart, and multi-instance active-active is forbidden. Real durability means the currently-dead Cloudflare lane or writing a persistence adapter — work, not a freebie.
  • Asymmetric abandonment. 1,697 test lines encode hard-won wire knowledge (form-encoding asymmetry, streaming sequences, the signature recipe). A rewrite discards proven verification for abstractions the project doesn't control.

Decided 2026-07-01: the product runs on Flue. The skeptic's brief above stands as the risk register, and his scorecard survives — no longer as a decision gate, but as the port's acceptance criteria. The port is gated by a behavioral parity harness: the existing tests' expectations, extracted into a lane-agnostic scenario suite, must pass on Flue before the custom harness is deleted.

OWNER DECISION · EXECUTED AS /GOAL G-PORT

The port is cheap for the same reason the spike would have been: the product's entire observable surface is Slack wire traffic — signed fixtures in, recorded Web API calls out — and both lanes speak exactly that surface. The tests' value is the encoded expectations, which port cleanly once decoupled from the lane-specific rigs. It is honest because its mandatory acceptance gates target exactly the skeptic's predicted failure points: restart durability, same-fixture provider swap, and a fully-offline run. The port ends with a dated record in docs/decisions/ — parity evidence, the exceptions file, and the Stage-4 artifacts.

    Context injection hardening (F14) — escape or structurally separate transcript rows from user text; consider per-row provenance markers. Channel members can currently forge context the model cannot distinguish from real messages.

/goal G-PORT

Port to Flue behind a behavioral parity harness, then delete the custom harness

time-boxed ≈2–3 weeks · Tier 2

Context. Two stacks exist: the working hand-rolled harness (src/server.ts → events-app.ts → slack-thread-runner.ts, with its own provider registry, session store, dedupe) and a vestigial Flue lane that throws at init (invalid model specifier). Decision: the product runs on Flue. The 46 tests drive signed Slack fixtures with an injected recording fetch — they encode the behavior to preserve. Flue supplies flue add channel slack, registerProvider() (incl. openai-compatible baseUrl registrations usable as a local offline stub), durable agents (db.ts), and flue dev --target node. Flue's Slack docs assign admission filtering and duplicate-claiming to the app, so turn-normalization.ts survives verbatim. Old lane frozen to bug-fixes-only for the duration.

  • Stage 0 — extract the contract. Lane-agnostic scenario suite: signed fixture(s) in → observables out (reply happens or nothing on the wire; count/target of final deliveries recorded by a fake-Slack backend; status set-then-cleared; reply-content properties; conversations.replies/history params; N duplicate events → 1 reply). Sources: the 46 tests + play-slack.md §5. Assertions behavioral, never byte-exact call sequences. Lane A adapter first. Verify: suite green on main; mutation check — deliberately break one behavior (e.g. skip the dedupe claim) and exactly the right scenario goes red.
  • Stage 1 — boot the Flue lane. Specifier → cloudflare-workers-ai/@cf/zai-org/glm-5.2 (re-confirm against the lockfile runtime); engines ≥22.18 + .nvmrc. Verify: a resolver-driving test is red on main, green after; flue:build green in CI on pinned Node.
  • Stage 2 — stand up the Flue Slack surface. flue add channel slack; port into the channel handler: admission (unchanged), duplicate claim keyed (channel, message ts) + event_id in an app-owned claim store, assignment/config, bounded hydration via @slack/web-api WebClient, status + chatStream. Slack API base URL injectable (slackApiUrl env; fallback: global-fetch preload). registerProvider: anthropic, cloudflare-workers-ai, local deterministic stub. Verify: flue dev --target node serves /channels/slack/events (signed → 200, tampered → 401, url_verification echoed); one full offline turn against the stub with network recorded/blocked — zero external traffic.
  • Stage 3 — parity. Lane B adapter: spawn the Flue app, WebClient → local fake Slack, quiescence-poll the wire log (dispatch is async). Verify: the entire scenario suite passes on Flue unmodified; intentional differences live in a reviewed per-scenario exceptions file — never silent assertion edits; CI runs both lanes during coexistence.
  • Stage 4 — Flue-only gains. Verify with committed evidence: (a) kill mid-conversation with file-backed db.ts → follow-up turn retains context, transcript attached; (b) same fixture answered via anthropic/… and cloudflare-workers-ai/…, both replies attached; (c) lookup_channel_brief as exactly one defineTool, invoked by the model, deny enforced with an honest message; (d) suite still green.
